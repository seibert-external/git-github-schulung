Du bist kein Programmierer? Perfekt. In dieser Session lernst du in Ruhe, was diese beiden Werkzeuge eigentlich sind – mit einfachen Bildern, Alltags­vergleichen und einer ersten eigenen Übung auf GitHub.

Kapitel 1

Das Problem: Arbeiten ohne Git

Bevor wir erklären, was Git ist, schauen wir, warum es überhaupt erfunden wurde. Wahrscheinlich kennst du das Chaos schon.

Stell dir vor, mehrere Leute arbeiten an derselben Datei – einem Angebot, einer Präsentation, einem Text. Man schickt sie per E-Mail hin und her, jeder speichert seine eigene Version, und am Ende heißt sie:

Typische Probleme dabei:

  • Niemand weiß sicher, welche Version die neueste ist.
  • Zwei Personen ändern gleichzeitig – eine Änderung überschreibt die andere.
  • Man kann nicht nachvollziehen, wer was wann und warum geändert hat.
  • Ein alter Stand lässt sich kaum sauber wiederherstellen.
⚠️ Häufiges Missverständnis

„Git ist nur was für Programmierer." – Nein. Git löst ein ganz allgemeines Problem: viele Versionen, viele Menschen, eine gemeinsame Wahrheit. Es wird vor allem für Programmcode benutzt, aber das Prinzip gilt für jede Art von Text­dateien.

🧑‍🏫 Trainer-Tipp

Lass die Teilnehmer kurz eigene Horror-Geschichten erzählen („zwei Leute im selben Excel…"). Das holt alle ab und macht sofort klar, wofür das Werkzeug gut ist – bevor ein einziger Fachbegriff fällt.

Kapitel 2

Die Lösung: Was ist „Versionsverwaltung"?

Eine Versionsverwaltung ist ein Werkzeug, das sich jede Version deiner Dateien merkt – und dazu, wer sie wann und warum geändert hat.

💡 Analogie

Denk an die Speicherstände in einem Videospiel oder an ein Logbuch: Du kannst jederzeit zu einem früheren Punkt zurückspringen, und es ist genau festgehalten, was passiert ist. Oder: wie „Änderungen nachverfolgen" in Word – aber für ein ganzes Projekt und für ein ganzes Team.

Mit so einem Werkzeug kannst du:

  • zu jedem alten Stand zurückspringen,
  • genau sehen, was sich von Stand zu Stand geändert hat,
  • und mit anderen arbeiten, ohne euch gegenseitig zu überschreiben.

Git ist das mit Abstand bekannteste Werkzeug dieser Art. Schauen wir es uns genauer an – und klären als Erstes den Unterschied, der am meisten Verwirrung stiftet.

📌 Merksatz

Versionsverwaltung = ein „Gedächtnis" für deine Dateien. Git ist das Werkzeug, das dieses Gedächtnis führt.

Exkurs · kleine Geschichte

Woher kommt Git eigentlich?

Git ist gar nicht so alt – und seine Entstehung ist eine kleine Notgeschichte. Sie hilft zu verstehen, warum das Werkzeug so funktioniert, wie es funktioniert.

Aus der Not geboren (2005)

Erfunden hat Git Linus Torvalds – derselbe Mann, der das Betriebssystem Linux gestartet hat (es läuft heute auf einem Großteil des Internets). Für die Weiterentwicklung von Linux arbeiteten tausende Programmierer weltweit zusammen. Dafür brauchten sie ein gutes Werkzeug – und genau das fiel 2005 plötzlich weg: Das bis dahin genutzte Programm durften sie nicht länger kostenlos verwenden, und die Alternativen waren für ein so riesiges Projekt schlicht zu langsam.

Also setzte sich Linus hin und baute in nur wenigen Wochen sein eigenes Werkzeug. Seine drei Ziele erklären bis heute, wie sich Git anfühlt:

  • Schnell – auch bei riesigen Projekten.
  • Für viele gleichzeitig – tausende Leute parallel, ohne sich zu blockieren.
  • Verteilt – jede Person hat die komplette Historie auf dem eigenen Rechner. Kein zentraler Server, der alles lahmlegt, wenn er ausfällt.
💡 Fun-Fact: der Name „Git"

„git" ist im britischen Englisch ein liebevoll-spöttisches Wort für einen sturen Dickkopf. Linus meinte selbstironisch, er benenne seine Projekte eben nach sich selbst – erst „Linux", dann „Git". 😄

Vom Geheimtipp zum Standard (2008 → heute)

Git war von Anfang an kostenlos und quelloffen – jeder durfte es nutzen. Das half ihm, sich überall zu verbreiten. 2008 kam dann GitHub dazu: eine Webseite, die auf Git aufbaut und es auch für Menschen ohne Kommandozeilen-Erfahrung benutzbar machte. Damit wurde Git endgültig populär – weit über Programmierer hinaus.

📌 Zum Schmunzeln & Merken

Wenn du heute GitHub benutzt, nutzt du Technik, die ursprünglich erfunden wurde, um tausende Linux-Freiwillige weltweit zu koordinieren. Ganz schön großer Ursprung für deine erste kleine Änderung.

🧑‍🏫 Trainer-Tipp

Bewusst kurz und locker halten – das ist die Auflockerung zwischendurch, kein Prüfungsstoff. Die „Linus = Linux = Git"-Verbindung und der Name-Gag kommen erfahrungsgemäß gut an und machen das abstrakte Thema menschlich. Wer mag, fragt: „Wer hat schon mal von Linux gehört?"

Kapitel 3

Der wichtigste Unterschied: Git ≠ GitHub

Diese beiden werden ständig verwechselt. Wenn du nur eine Sache aus dieser Schulung mitnimmst, dann diese:

GitGitHub
Was ist es?Ein Werkzeug / ProgrammEine Webseite / Plattform
Wo läuft es?Auf deinem eigenen RechnerIm Internet (in der „Cloud")
Internet nötig?Nein, geht auch offlineJa, es ist eine Webseite
Wofür?Versionen merkenProjekte teilen & gemeinsam dran arbeiten
💡 Analogie

Es ist wie bei E-Mail und Gmail: „E-Mail" ist die Technik, „Gmail" ist nur ein Anbieter, bei dem du sie nutzt. Genauso ist Git die Technik – und GitHub einer von mehreren Anbietern. Andere Anbieter sind z. B. GitLab oder Bitbucket.

📌 Merksatz

Git = das Werkzeug auf deinem Rechner. GitHub = der gemeinsame Treffpunkt im Internet, wo das Projekt liegt.

🧑‍🏫 Trainer-Tipp

Die E-Mail/Gmail-Analogie zündet bei Nicht-Devs fast immer. Wiederhol sie ruhig später nochmal. Wenn jemand fragt „brauche ich beides?": In dieser Firma arbeitet ihr meist über GitHub im Browser – Git läuft im Hintergrund mit, ohne dass man es direkt sieht.

Kapitel 4

Das Repository – kurz „Repo"

Ein Repository ist einfach dein Projektordner – aber mit Gedächtnis. Es enthält deine Dateien und die komplette Geschichte aller Änderungen.

Wenn du auf GitHub ein neues Projekt anlegst, legst du genau so ein Repository an. Es kann sein:

  • privat – nur du (und eingeladene Personen) seht es, oder
  • öffentlich – jeder im Internet kann es ansehen.
📌 Merksatz

Ein Repository ist ein Projektordner, der sich seine komplette Vergangenheit merkt. „Lokal" = bei dir, „Remote" = auf GitHub.

Kapitel 5

Der Commit – ein gespeicherter Schnappschuss

Ein Commit ist ein gespeicherter Stand deines Projekts zu einem bestimmten Moment – zusammen mit einer kurzen Notiz, was du geändert hast und warum.

💡 Analogie

Ein Commit ist wie ein Foto mit Bildunterschrift: Es hält den Moment fest und sagt in einem Satz, was darauf passiert. Diese Bildunterschrift nennt man die Commit-Nachricht.

Die Aneinanderreihung aller Commits ist die Historie deines Projekts. Du kannst sie jederzeit durchblättern wie ein Tagebuch – und sehen, wer wann was gemacht hat.

⚠️ Häufiges Missverständnis

Ein Commit ist nicht dasselbe wie „Datei speichern". Speichern macht dein Programm sowieso ständig. Ein Commit ist ein bewusster Schnappschuss, den du mit einer Nachricht festhältst – etwa „so, dieser Abschnitt ist fertig".

📌 Merksatz

Commit = ein bewusst gespeicherter Stand + eine kurze Nachricht „was & warum".

🧑‍🏫 Trainer-Tipp

Betone die Nachricht. Gute Commit-Nachrichten sind das, was Historie überhaupt lesbar macht. Gegenbeispiel zum Schmunzeln: Commits mit der Nachricht „asdf" oder „Änderung" – damit kann später niemand etwas anfangen.

Kapitel 6 · Kernmodell

Die drei Bereiche: vom Schreibtisch ins Fotoalbum

Damit aus einer Änderung ein Commit wird, durchläuft sie drei Stationen. Das ist das Herzstück von Git. Wir gehen es ganz langsam durch.

Schritt für Schritt

  1. Arbeitsverzeichnis (Schreibtisch): Hier bearbeitest du deine Dateien ganz normal.
  2. Staging / Sammelkorb: Du legst gezielt die Änderungen hinein, die in den nächsten Schnappschuss sollen. (So kannst du z. B. drei von fünf Änderungen festhalten und zwei noch liegen lassen.)
  3. Repository / Fotoalbum: Mit dem Commit wird aus dem Korb ein fester Schnappschuss in der Historie.
💡 Analogie

Ein Paket packen: Auf dem Schreibtisch liegen viele Dinge (Arbeitsverzeichnis). Du legst ausgewählte Dinge in den Karton (Staging). Dann klebst du den Karton zu und schickst ihn ins Archiv – fertig, nicht mehr änderbar (Commit ins Repository).

⚠️ Gut zu wissen für GitHub im Browser

Wenn du direkt auf der GitHub-Webseite arbeitest, passieren „Sammelkorb" und „Foto machen" in einem Schritt: Du klickst „Commit changes" und gibst die Nachricht ein. Das Modell mit den drei Bereichen läuft trotzdem im Hintergrund – gut, es zu kennen, damit dich später kein Begriff überrascht.

📌 Merksatz

ändern ● → add in den Sammelkorb ● → commit ins Album ●.

🧑‍🏫 Trainer-Tipp

Das ist das schwierigste Kapitel für Einsteiger – nimm dir hier am meisten Zeit. Zeig die Paket-Analogie mit den Händen. Wer das Staging einmal verstanden hat, versteht später Push/Pull fast von allein.

Kapitel 7

Branches – parallel arbeiten ohne Chaos

Ein Branch (deutsch: „Zweig") ist eine parallele Arbeitskopie. Du kannst darin etwas ausprobieren, ohne die offizielle Version kaputtzumachen.

💡 Analogie

Wie ein Sandkasten oder ein Paralleluniversum: Du baust in Ruhe an deiner Idee. Klappt sie, holst du sie zurück in die Hauptversion. Klappt sie nicht, wirfst du den Zweig einfach weg – die Hauptversion hat nie etwas davon mitbekommen.

Der Hauptzweig heißt fast immer main (früher oft master) – das ist die „gilt für alle"-Version. Branches benutzt man, um:

  • eine neue Idee oder ein Feature in Ruhe auszuprobieren,
  • mit mehreren Leuten gleichzeitig an verschiedenen Dingen zu arbeiten,
  • die Hauptversion immer funktionsfähig zu halten.
📌 Merksatz

Ein Branch ist ein sicherer Spiel-Zweig neben der Hauptversion main. Experimentieren erlaubt – ohne Risiko.

Kapitel 8

Merge – Zweige wieder zusammenführen

Mergen heißt: die Änderungen aus einem Branch zurück in einen anderen (meist main) holen. Aus zwei Linien wird wieder eine.

Und was ist ein „Merge-Konflikt"?

Meistens klappt das Zusammenführen ganz automatisch. Ein Konflikt entsteht nur, wenn zwei Leute genau dieselbe Zeile unterschiedlich geändert haben. Dann kann Git nicht raten, welche Version stimmt – und fragt einen Menschen: „Welche Variante soll gelten?"

⚠️ Keine Angst vor Konflikten

Ein Merge-Konflikt ist völlig normal und kein Fehler. Git zeigt dir genau die Stelle und beide Varianten; du wählst aus oder kombinierst sie. Danach ist alles wieder gut.

📌 Merksatz

Merge = Zweige vereinen. Konflikt = dieselbe Stelle doppelt geändert, ein Mensch entscheidet.

Kapitel 9 · Zusammenarbeit

Clone, Push und Pull

Jetzt verbinden wir „bei dir" (lokal) mit „auf GitHub" (remote). Dafür gibt es genau drei Worte, die du dir merken solltest.

WortBedeutungIn einem Satz
CloneErste Kopie holen„Ich hole mir das Projekt von GitHub das erste Mal auf meinen Rechner."
PushHochladen„Ich schicke meine Commits hinauf zu GitHub, damit andere sie haben."
PullHerunterladen„Ich hole mir die neuesten Änderungen der anderen von GitHub herunter."
📌 Merksatz

Push = hoch (geben), Pull = runter (holen), Clone = die allererste Kopie.

🧑‍🏫 Trainer-Tipp

Eselsbrücke laut mitsprechen lassen: „Pushen heißt drücken – nach oben. Pullen heißt ziehen – zu mir herunter." Bei reiner Browser-Arbeit auf GitHub macht man Push/Pull oft gar nicht selbst – das ist trotzdem das Vokabular, das überall fällt.

Kapitel 10 · Der wichtigste GitHub-Ablauf

Pull Request & Review – so arbeitet ein Team

Das ist der Ablauf, den du bei GitHub am häufigsten sehen wirst. Ein Pull Request (kurz „PR") heißt im Grunde: „Ich schlage diese Änderung vor – schaut sie euch bitte an, bevor sie offiziell wird."

💡 Analogie

Wie „Änderungen vorschlagen" in Google Docs: Du änderst nicht einfach wild im Original, sondern machst einen Vorschlag. Eine Kollegin schaut drauf, kommentiert, und erst wenn alle einverstanden sind, wird der Vorschlag übernommen.

Im Pull Request sieht man genau, was sich ändert (alte vs. neue Zeilen, grün/rot markiert), kann darunter diskutieren, Änderungen anfordern und am Ende auf „Merge" klicken.

Und „Issues"?

Ein Issue ist einfach ein Ticket / eine Notiz auf GitHub: „Hier ist ein Fehler", „das wünschen wir uns", „das ist noch zu tun". Issues sind die To-do-Liste des Projekts – Pull Requests sind die konkreten Lösungen dazu.

📌 Merksatz

Pull Request = ein Änderungs-Vorschlag, der erst geprüft und dann übernommen (gemergt) wird. Issue = ein Ticket auf der To-do-Liste.

🧑‍🏫 Trainer-Tipp

Das ist der praxisrelevanteste Punkt für Nicht-Devs – hier verbringen sie später ihre Zeit. Wenn möglich: zeig live einen echten (harmlosen) Pull Request im Browser, damit sie die grün/rote Diff-Ansicht und den „Merge"-Button einmal gesehen haben.

Kapitel 11

Die Kommandozeile – der „Profi-Weg"

Alles, was du im Browser klickst, geht auch über getippte Befehle in einem schwarzen Fenster – der Kommandozeile (auch „Terminal"). Du musst das nicht können, um loszulegen. Aber du sollst wissen, dass es das gibt.

Terminal
# So sieht ein typischer Arbeitsablauf in der Kommandozeile aus:
$ git clone https://github.com/firma/projekt.git   # erste Kopie holen
$ git add angebot.txt        # Änderung in den Sammelkorb legen
$ git commit -m "Preise aktualisiert"   # Schnappschuss + Nachricht
$ git push                   # zu GitHub hochladen

Du erkennst die Begriffe wieder – es sind genau dieselben wie im Browser, nur getippt:

Im Browser auf GitHub …… als Befehl in der Kommandozeile
Projekt auf den Rechner holengit clone …
Änderung markieren (Sammelkorb)git add datei
„Commit changes" klickengit commit -m "Nachricht"
Hochladengit push
Neueste Änderungen holengit pull
Neuen Branch anlegengit branch name / git switch -c name
Zusammenführengit merge name
⚠️ Wichtig zur Beruhigung

Die Kommandozeile sieht „nach Hacker" aus, ist aber nur eine andere Bedienoberfläche für genau dieselben Dinge. Für den Anfang reicht der Browser völlig. Viele nutzen außerdem grafische Programme wie GitHub Desktop – Buttons statt Tippen.

📌 Merksatz

Browser, GitHub Desktop, Kommandozeile – drei Wege, dasselbe zu tun. Die Begriffe bleiben überall gleich.

Kapitel 12 · Jetzt du!

✋ Mitmach-Teil: deine erste Änderung auf GitHub

Komplett im Browser, ohne Installation. Am Ende hast du ein eigenes Repository angelegt und deinen ersten Commit gemacht. Folge den Schritten – ich begleite dich.

✋ Was du brauchst

Nur einen Browser und ein kostenloses GitHub-Konto. Falls noch keins da ist: auf github.com → „Sign up" → mit E-Mail registrieren. (In der Firma evtl. das vorhandene Organisations-Konto nutzen – frag im Zweifel deine Trainerin / deinen Trainer.)

  1. Repository anlegen. Oben rechts auf ＋ → „New repository". Gib einen Namen ein (z. B. mein-erstes-repo), setze einen Haken bei „Add a README file" und klicke „Create repository". → Du hast gerade ein Repository erstellt (Kapitel 4).
  2. Die README öffnen & bearbeiten. Klicke auf die Datei README.md und dann oben rechts auf das Stift-Symbol ✏️ („Edit"). Schreib ein paar Zeilen über dich oder das Projekt.
  3. Deinen ersten Commit machen. Klicke oben rechts auf „Commit changes…". Gib eine kurze Nachricht ein, z. B. „README mit Begrüßung ergänzt", und bestätige. → 🎉 Glückwunsch, das war dein erster Commit – ein Schnappschuss mit Nachricht (Kapitel 5)!
  4. Die Historie ansehen. Geh zurück zur Startseite des Repos und klicke auf „Commits" (Uhr-/Listensymbol über der Dateiliste). Dort siehst du deinen Schnappschuss mit Nachricht, Autor und Zeitpunkt. → Das ist die Historie aus Kapitel 2.

Bonus (wenn Zeit ist): ein Pull Request

  1. Branch anlegen. Oben links über der Dateiliste auf das Branch-Menü (main) klicken, einen neuen Namen tippen (z. B. kleine-aenderung) und „Create branch" wählen. → ein Branch (Kapitel 7).
  2. Im Branch ändern & committen. README erneut bearbeiten und committen – diesmal landet der Commit im neuen Branch, nicht in main.
  3. Pull Request öffnen. GitHub schlägt oben „Compare & pull request" vor → anklicken, kurz beschreiben, „Create pull request". → dein Pull Request (Kapitel 10).
  4. Mergen. Auf „Merge pull request" klicken → deine Änderung ist jetzt offiziell in main. Fertig! 🥳
📌 Geschafft

Du hast ein Repository angelegt, committet, die Historie gelesen und (optional) einen Pull Request gemergt – den kompletten GitHub-Grundablauf einmal selbst durchgespielt.

🧑‍🏫 Trainer-Tipp

Mach Schritt 1–4 einmal vor (Beamer), dann machen alle parallel mit. Plane Puffer fürs Login/Sign-up ein – das kostet bei Einsteigern erfahrungsgemäß die meiste Zeit. Den Bonus nur angehen, wenn die Gruppe sicher wirkt.

Kleiner Selbsttest

Hast du's? – 4 Fragen zum Aufklappen

Erst überlegen, dann aufklappen. Keine Noten – nur zum Festigen. 🙂

Was ist der Unterschied zwischen Git und GitHub?
Git ist das Werkzeug auf deinem Rechner (merkt sich Versionen, geht offline). GitHub ist eine Webseite im Internet, auf der Projekte liegen und Teams zusammenarbeiten – wie E-Mail (Technik) vs. Gmail (Anbieter).
Was ist ein Commit?
Ein bewusst gespeicherter Schnappschuss deines Projekts zu einem Moment – zusammen mit einer kurzen Nachricht, was & warum geändert wurde. Wie ein Foto mit Bildunterschrift.
Push oder Pull – was ist was?
Push = hochladen (deine Commits hinauf zu GitHub geben). Pull = herunterladen (die neuesten Änderungen der anderen zu dir holen). Clone = die allererste Kopie holen.
Wofür ist ein Pull Request gut?
Um eine Änderung vorzuschlagen, bevor sie offiziell wird: andere können sie ansehen, kommentieren und freigeben – dann wird sie in main gemergt. Wie „Änderungen vorschlagen" in Google Docs.
